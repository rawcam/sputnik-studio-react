## 1. Цель и задачи

**Цель** — создать современную, модульную и расширяемую веб-платформу для проектирования AV-систем, заменив монолитную версию v7 на React-приложение с устранением архитектурных проблем (скрытые контейнеры, конфликты стилей, сложность поддержки).

**Задачи**:
1. Перенести весь функционал v7 (проекты, тракты, калькуляторы, шаблоны) на React с TypeScript.
2. Внедрить роутинг (React Router) для полной изоляции разделов.
3. Использовать Redux Toolkit для единого состояния.
4. Перейти на IndexedDB (Dexie) для локального хранения данных.
5. Разработать компонентную архитектуру, облегчающую дальнейшее расширение (графический редактор, финансовая аналитика).
6. Обеспечить полную стилистическую совместимость с предыдущей версией (светлая/тёмная темы, стеклянные карточки, адаптивность).
7. Внедрить **ролевую модель** и **финансовую аналитику** (маржинальность, рентабельность, кассовые разрывы, общехозяйственные расходы).
8. Ввести **цифровой идентификатор проекта** (4 цифры) с разбивкой по категориям.

## 3. Функциональные требования

### 3.1. Общая структура
- **Роутинг**: четыре основные страницы:
  - `/dashboard` — главная панель управления (виджеты, карусель активных проектов)
  - `/projects` — управление проектами (список, создание, редактирование)
  - `/calculations` — все калькуляторы и работа с трактами (сайдбар + основная область)
  - `/templates` — конструктор документов (шаблоны)
- **Сайдбар** показывается **только** на странице `/calculations`. В нём расположены аккордеоны с настройками (видео, сеть, LED, звук, ВКС, эргономика, тракты, статистика, энергия, управление).

### 3.2. Роли пользователей и авторизация
- **Роли**:
  - `director` — генеральный директор: полный доступ ко всем данным, включая общехозяйственные расходы и финансовую отчётность компании.
  - `pm` (руководитель проектов / ГИП) — видит все проекты, финансовые показатели по проектам (маржу, рентабельность, кассовые разрывы по проекту), но не общехозяйственные расходы.
  - `engineer` — видит только технические детали проектов, тракты, калькуляторы, загрузку по себе. Финансы скрыты.
  - `designer` — аналогично инженеру, может редактировать тракты и схемы.
  - `logist` — видит закупки, сроки поставок, может влиять на расходную часть (закупки), но не на бюджет продаж.
- **Аутентификация** на первом этапе: локальное хранение роли (можно переключать для тестирования). В будущем — интеграция с внешней системой (JWT).
- **Интерфейс**: компоненты проверяют роль и скрывают/показывают элементы; топбар отображает имя и роль, кнопка переключения роли (для демо).

### 3.3. Управление проектами
- CRUD проектов.
- **Цифровой идентификатор** (`shortId`): четырёхзначное число, генерируется автоматически при создании проекта с учётом выбранной категории.
- **Категории проекта** (диапазоны `shortId`):
  - `new` (0000–1999) — новый проект с нуля
  - `modernization` (2000–3999) — модернизация существующей системы
  - `service` (4000–5999) — сервисное обслуживание
  - `standard` (6000–7999) — типовой проект
  - `pilot` (8000–9999) — пилотный / НИОКР
- Поля: название, бюджет (сумма контракта), инженер, руководитель проекта, дата начала, статус (пресейл → стадия П → стадия Р → монтаж → завершён), категория.
- **Финансовая модель**:
  - Доходная часть: `contractAmount`, `incomeSchedule` (массив дат и сумм)
  - Расходная часть: `expenseSchedule` (массив дат, сумм и типов)
  - Фактические доходы/расходы: `actualIncome`, `actualExpenses` (вычисляются из расписаний с учётом статуса оплаты)
- **Дорожная карта** в двух вариантах: плановая и фактическая (с возможностью сравнения).
- Встречи, закупки, сервисные выезды (см. п. 3.6).

### 3.4. Финансовая модель проекта
- **Маржа и рентабельность**:
  - Плановая маржа = `contractAmount` – сумма плановых расходов.
  - Плановая рентабельность = плановая маржа / `contractAmount`.
  - Фактическая маржа = `actualIncome` – `actualExpenses`.
  - Фактическая рентабельность = фактическая маржа / `actualIncome` (или / `contractAmount` при завершении).
- **Кассовый разрыв по проекту**: накопленный денежный поток на основе `incomeSchedule` и `expenseSchedule`. Визуализация в детальном отчёте.

### 3.5. Общехозяйственные расходы
- Отдельный слайс `companyExpenses` для расходов, не привязанных к проекту (аренда офиса, налоги, зарплата административного персонала и т.п.).
- Доступны только для роли `director`.
- Влияют на общий кассовый разрыв компании.

### 3.6. Сервисные и регламентные работы
- В каждом проекте можно создать список сервисных выездов / регламентных работ:
  - Поля: дата, тип работ, статус (запланировано, выполнено, отменено), ответственный, стоимость (опционально).
- На дашборде (виджет «Сервис и регламент») отображаются ближайшие выезды (по ролям: директор/ГИП видят всё, инженер — только свои, логист — все).
- Клик по виджету ведёт на страницу «Сервисное расписание» с фильтрацией.

### 3.8. Виджеты дашборда (с учётом ролей)
Сетка из 5 колонок. Содержимое зависит от роли.

| Колонка | Директор | ГИП | Инженер/Проектировщик | Логист |
|---------|----------|-----|----------------------|--------|
| 1. Финансы компании | Общая выручка, маржа, рентабельность, кассовый разрыв | Скрыт / свёрнут в общую цифру | Не видит | Видит закупки и расходы по проектам |
| 2. Финансы проектов | Топ‑3 по марже, проекты с отрицательной маржой, разбивка по категориям | Сводка по всем проектам (маржа, рентабельность) | Не видит | Не видит маржу (видит закупки и статус оплат) |
| 3. Сервис и регламент | Видит всё | Видит свои объекты | Видит только свои объекты (по привязке) | Видит плановые выезды (логистика) |
| 4. Загрузка | Видит всех | Видит всех | Видит свою | Не видит (или видит косвенно) |
| 5. Риски | Финансовые риски (кассовый разрыв, низкая маржа) | Технические + финансовые по своим проектам | Технические риски (латентность, PoE) | Риски по срокам закупок |

## 5. Критерии приемки (дополненные)
- Все роли отображают только разрешённые им данные.
- Финансовые расчёты (маржа, рентабельность, кассовый разрыв) корректны.
- Общехозяйственные расходы учитываются в отчётах компании.
- Сервисные работы создаются, редактируются, отображаются в виджете и на отдельной странице.
- Цифровой ID генерируется в соответствии с выбранной категорией, уникален.
- Кнопка «Новый тракт» находится вверху контейнера (проблема v7 решена).
